# Verification Skill

在代码变更后自动验证正确性。

## 触发场景

1. **用户要求验证**：用户说"验证一下"、"检查代码"
2. **自动触发**：检测到代码变更后提示用户是否验证

## 执行步骤

### 1. 识别变更

通过 `sessions_history` 获取最近的 tool calls，识别：
- `write` / `edit` 操作的文件
- `exec` 运行的命令（如 git diff）

### 2. 确定验证范围

根据变更类型选择验证方式：

| 变更类型 | 验证方式 |
|---|---|
| 配置文件 | 语法检查、schema 验证 |
| 测试文件 | 运行测试 `exec npm test` |
| 构建脚本 | 运行构建 `exec npm run build` |
| API 代码 | 运行相关测试或集成测试 |
| 样式/前端 | 视觉检查（截图）或构建验证 |

### 3. 执行验证

```bash
# 通用验证
- 语法检查：<对应的 lint 命令>
- 单元测试：<对应的 test 命令>
- 构建验证：<对应的 build 命令>

# 如果变更涉及文件
- 读取变更文件
- 检查是否有明显错误
```

### 4. 报告结果

```markdown
## 验证结果

### 变更文件
- <文件1>
- <文件2>

### 验证执行
- <命令1>: ✅ 通过 / ❌ 失败
- <命令2>: ✅ 通过 / ❌ 失败

### 问题（如果有）
<具体错误信息和位置>

### 建议
<修复建议>
```

## 限制

- 验证不是万能的，只能检查可自动化的部分
- 复杂逻辑仍需人工 review
- 如果验证失败，明确告知用户需要人工介入
